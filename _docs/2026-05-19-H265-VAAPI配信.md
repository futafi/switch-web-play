# H.265 VA-API ハードウェアエンコード配信 (2026-05-19)

GStreamer コンテナを `restreamio/gstreamer:latest-dev` ベースから Debian 12 ベースのカスタムイメージへ切り替え、Intel VA-API による H.265 ハードウェアエンコードを有効化した。

## 変更内容

### Docker イメージ

`streaming/Dockerfile` を追加した。

- ベース: `debian:12`
- VA-API: `libva2`, `libva-drm2`, `intel-media-va-driver-non-free`
- GStreamer: `gstreamer1.0-vaapi`, `gstreamer1.0-rtsp`, `gstreamer1.0-alsa`, base/good/bad/ugly/libav
- gstd: RidgeRun `gstd-1.x` をイメージ内でビルド
- 環境変数:
  - `LIBVA_DRIVER_NAME=iHD`
  - `GST_VAAPI_ALL_DRIVERS=1`

`intel-media-va-driver-non-free` は Debian の `non-free` コンポーネントにあるため、Dockerfile 内で apt sources に `contrib non-free non-free-firmware` を追加している。

### GStreamer パイプライン

`streaming/entrypoint.sh` の映像エンコードを `x264enc` から `vaapih265enc` へ変更した。

```text
v4l2src
! image/jpeg,width=1920,height=1080,framerate=30/1
! jpegdec
! videoconvert
! videoscale
! capsfilter name=scaler caps=video/x-raw,width=${WIDTH},height=${HEIGHT}
! vaapih265enc name=encoder tune=low-power rate-control=cbr bitrate=${BITRATE} keyframe-period=30
! h265parse config-interval=1
! rtspclientsink location=rtsp://${RTSP_HOST:-mediamtx}:8554/cam name=sink
```

低遅延向けに以下を維持した。

- `tune=low-power`
- `rate-control=cbr`
- `keyframe-period=30`
- `max-bframes` はデフォルトが `0`
- `h265parse config-interval=1` で VPS/SPS/PPS を定期挿入

gstd/API 互換のため、動的変更対象の要素名は従来通り `encoder` と `scaler` のままにしている。`vaapih265enc` の `bitrate` も kbps 指定なので、既存 UI の 12000/8000/4000/2000 はそのまま使える。

### docker compose

`docker-compose.yml` と `docker-compose.host.yml` の gstreamer service を `streaming/Dockerfile` build に変更し、GPU render node を渡すようにした。

```yaml
devices:
  - ${VIDEO_DEVICE}:/dev/video0
  - /dev/dri/renderD128:/dev/dri/renderD128
  - /dev/snd:/dev/snd
```

## 検証結果

ビルド:

```bash
docker compose build gstreamer
```

VA-API:

```bash
docker run --rm \
  --device /dev/dri/renderD128:/dev/dri/renderD128 \
  --entrypoint vainfo web-switch-gstreamer \
  --display drm --device /dev/dri/renderD128
```

確認結果:

- VA-API version: `1.17`
- Driver: `Intel iHD driver ... 23.1.1`
- `VAProfileHEVCMain: VAEntrypointEncSlice`
- `VAProfileHEVCMain: VAEntrypointEncSliceLP`

GStreamer:

```bash
docker run --rm \
  --device /dev/dri/renderD128:/dev/dri/renderD128 \
  --entrypoint gst-launch-1.0 web-switch-gstreamer \
  -q videotestsrc num-buffers=30 is-live=true \
  ! video/x-raw,width=1280,height=720,framerate=30/1 \
  ! videoconvert \
  ! vaapih265enc tune=low-power rate-control=cbr bitrate=4000 keyframe-period=30 \
  ! h265parse config-interval=1 \
  ! fakesink
```

結果: H.265 ハードウェアエンコードの短時間パイプラインは成功。

実配信:

```bash
docker compose up -d mediamtx gstreamer
docker compose logs --no-color --tail=160 mediamtx gstreamer
```

mediamtx ログで以下を確認した。

```text
[path cam] stream is available and online, 2 tracks (Opus, H265)
[RTSP] [session ...] is publishing to path 'cam'
```

gstd ログで以下を確認した。

```text
GstVaapiEncodeH265 encoder
bitrate=12000
rate-control=cbr
tune=low-power
state: playing
```

## 注意点

`gst-inspect-1.0 vaapih265enc` は GPU デバイスを渡さないコンテナでは `No such element` になる。VA-API プラグインは実デバイス検出後に `vaapih265enc` を登録するため、確認時は `--device /dev/dri/renderD128:/dev/dri/renderD128` を付ける。

mediamtx 側は H.265 + Opus のパスを WebRTC listener 有効状態で受け取れている。実視聴はブラウザ側の WebRTC H.265 対応に依存するため、ブラウザで映像が出ない場合はクライアントの H.265/HEVC 対応状況を確認する。
