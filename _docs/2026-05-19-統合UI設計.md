## 映像表示: WHEP クライアント

mediamtxのWHEP APIを使ってWebRTC映像を取得する。

### WHEPハンドシェイク手順

1. `POST http://<host>:8889/cam/whep` に SDP offer を送信
   - `Content-Type: application/sdp`
   - Body: RTCPeerConnectionで生成した offer SDP
2. レスポンス: `201 Created`
   - Body: answer SDP
   - `Location` ヘッダ: セッションURL（切断時にDELETE）
3. ICE candidate:
   - `PATCH <Location URL>` で追加候補を送信（Trickle ICE）
   - `Content-Type: application/trickle-ice-sdpfrag`

### JavaScript実装の要点

```javascript
async function startVideo(videoElement) {
  const pc = new RTCPeerConnection();
  pc.addTransceiver('video', { direction: 'recvonly' });
  pc.addTransceiver('audio', { direction: 'recvonly' });

  pc.ontrack = (ev) => {
    videoElement.srcObject = ev.streams[0];
  };

  const offer = await pc.createOffer();
  await pc.setLocalDescription(offer);

  // mediamtxのWHEPエンドポイント
  const whepUrl = `http://${location.hostname}:8889/cam/whep`;
  const res = await fetch(whepUrl, {
    method: 'POST',
    headers: { 'Content-Type': 'application/sdp' },
    body: pc.localDescription.sdp,
  });

  const answerSdp = await res.text();
  await pc.setRemoteDescription({ type: 'answer', sdp: answerSdp });

  // セッションURL（切断時にDELETE）
  const sessionUrl = res.headers.get('Location');
  return { pc, sessionUrl };
}
```

注: `location` ヘッダが相対パスの場合はベースURL結合が必要。mediamtxのWHEPはTrickle ICEなしでも動作する（offer時にICE候補を全て含める場合）。

```

│                                                          │
│  [ZL]                                          [ZR]      │
│  [L]                                           [R]       │
│                                                          │
│                                                          │
│       ◉            [-]  [H]  [+]            X           │
│    (L Stick)                             Y     A         │
│                                             B            │
│                                                          │
│    ┌─┬─┬─┐                               ◉             │
│    │←│↑│→│          [CAP]             (R Stick)          │

│                                                          │
└──────────────────────────────────────────────────────────┘

左側:
  - 上段: ZL, L（画面上端付近）
  - 中段: 仮想左スティック（タッチドラッグで傾き、円形エリア）
  - 下段: D-pad (Hat)

中央:
  - Minus, Home, Plus（横並び）
  - Capture（下側）

右側:
  - 上段: ZR, R（画面上端付近）
  - 中段: X/Y/A/Bフェイスボタン（ダイヤモンド配置）
  - 下段: 仮想右スティック
```

### オーバーレイの表示ルール

- **デフォルト非表示**（物理ゲームパッド or キーボード使用想定）
- **⚙ アイコン** からトグルで表示/非表示
- **物理ゲームパッド接続検出時**: オーバーレイを自動非表示（⚙アイコンは残る）
- **表示中**: 半透明（ユーザー調整可能、初期値50%）
- **タッチ領域**: ボタン画像より約1.5倍大きく取る（押しやすさ優先）

### 仮想スティックの動作

- タッチ開始点を中心として扱う（固定位置ではなく、触れた場所が中心になる）
- ドラッグ距離に応じて0-255のスティック値を算出
- 最大ドラッグ半径を超えた場合は255（最大傾き）にクランプ
- タッチ終了で128（中央）に戻す

### 解像度・ビットレート変更

解像度/ビットレート変更にはGStreamerパイプラインの再起動が必要。

**サーバー側API** (`nxmc-server`に追加):

```
POST /api/stream
Content-Type: application/json

{
  "width": 1280,
  "height": 720,
  "bitrate": 8000
}
```

**サーバーの処理**:
1. リクエストを受け取る
2. `docker compose` 経由でgstreamerコンテナを環境変数付きで再起動:
   ```
   WIDTH=1280 HEIGHT=720 BITRATE=8000 docker compose restart gstreamer
   ```
   または `docker compose up -d gstreamer` で環境変数を反映
3. ブラウザ側でWHEP再接続（映像が途切れるため）

**解像度プリセット**:

| 名前 | width | height | デフォルトbitrate |
|------|-------|--------|------------------|
| 1080p | 1920 | 1080 | 12000 |
| 720p | 1280 | 720 | 8000 |
| 480p | 854 | 480 | 4000 |
| 360p | 640 | 360 | 2000 |

**ビットレートプリセット** (kbps):
- 12000, 8000, 4000, 2000

注: HDMIキャプチャ（MS2109）の対応解像度は `/dev/video0` のcapsに依存する。1080pと720pは確認済み（`_docs/2026-05-18-映像配信テスト.md`）。480pと360pはv4l2srcでのスケーリングが必要になる可能性がある（`videoscale` エレメントを挿入）。

## 入力系の移植

現在の `nxmc/cmd/nxmc-server/web/index.html` の入力ロジックをそのまま流用する。

### 維持するもの

- **キーボードマッピング** (`KEY_MAP`): 変更なし
- **Gamepad APIマッピング** (`GP_BUTTON_MAP`): 変更なし（[17]のCapture対応含む）
- **入力ループ**: `requestAnimationFrame` で60fpsポーリング、変化時のみWebSocket送信
- **WebSocket プロトコル**: `{"buttons":4,"hat":8,"lx":128,"ly":128,"rx":128,"ry":128}`
- **deadzone**: 0.15（アナログスティック）

### 追加するもの

- **タッチ入力**: `touchstart`/`touchmove`/`touchend` イベントからボタン・スティック状態を生成
  - マルチタッチ対応（左手スティック + 右手ボタン同時操作）
  - 各タッチを `touch.identifier` で追跡
  - タッチ領域のヒットテストでどのボタン/スティックか判定
- **入力マージ**: キーボード・ゲームパッド・タッチの3入力をOR合成してレポート生成

## 実装順序

1. **統合UIのHTML/CSS/JS** — 映像表示 + オーバーレイ + 設定パネル
   - まずWHEP映像表示を動かす
   - 既存の入力ロジックを移植
   - タッチオーバーレイを実装
   - 設定パネルを実装
2. **nxmc-server拡張** — 解像度/ビットレート変更API
3. **Dockerfile** — nxmc-serverのコンテナ化
4. **docker-compose.yml** — 全サービス統合
5. **解像度変更の検証** — 480p/360pがv4l2srcで対応できるか確認、必要なら`videoscale`挿入
