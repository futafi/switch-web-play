# 映像配信テスト (2026-05-18)

HDMIキャプチャの映像・音声をmediamtx経由でWebRTCブラウザ視聴できるか検証した。

## ホスト環境

- CPU: 13th Gen Intel Core i7-1360P
- iGPU: Intel Iris Xe Graphics (Raptor Lake)
- OS: Linux (Debian 12系)
- HDMIキャプチャ: MacroSilicon MS2109
  - 映像: `/dev/video0` — MJPEG/YUYV422, 最大1920x1080
  - 音声: ALSA `hw:2,0` — PCM 16bit 48kHz stereo

## 構成

```
HDMIキャプチャ ─→ エンコーダ(ffmpeg/GStreamer) ─RTSP→ mediamtx ─WebRTC→ ブラウザ
                  /dev/video0 + hw:2,0              :8554         :8889
```

- mediamtx: `bluenviron/mediamtx:latest` (v1.18.2) — WebRTC配信サーバー
- エンコーダとmediamtxは別コンテナで動作 (Docker Compose, `network_mode: host`)
- ブラウザから `http://<ホストIP>:8889/cam` でアクセス

## 実験1: ffmpeg

- イメージ: `linuxserver/ffmpeg:latest`
- 映像: MJPEG → libx264 (ultrafast, zerolatency)
- 音声: PCM → libopus 128kbps
- 結果: **映像・音声ともに動作確認済み**

注意点:
- `bluenviron/mediamtx` コンテナにはffmpegが入っていないため、`runOnInit` で起動する方式は使えない。別コンテナが必要。
- 音声コーデックはWebRTCがAACに非対応のため、Opusを使う必要がある (`skipping track (MPEG-4 Audio)` エラーになる)

## 実験2: GStreamer (ソフトウェアエンコード)

- イメージ: `restreamio/gstreamer:latest-dev`
  - `latest` タグは存在しない。`latest-dev`, `latest-prod` 等を使う必要がある
  - `-dev` 版にはrtspclientsink, x264enc, opusenc, alsasrc すべて含まれる
- パイプライン: `v4l2src → jpegdec → videoconvert → x264enc → rtspclientsink`
- 映像: MJPEG → x264enc (ultrafast, zerolatency, 12Mbps, keyframe毎秒)
- 音声: ALSA → opusenc (128kbps, frame-size=10ms)
- 結果: **映像・音声ともに動作確認済み**

所感:
- 4Mbpsだと動きの激しいシーンでブロックノイズが出る。12Mbpsで改善。

## 実験3: GStreamer (VAAPI ハードウェアエンコード) — 失敗

ホストのVA-API環境:
- ドライバ: Intel iHD v23.1.1 (VA-API 1.17)
- H264エンコード対応: `VAProfileH264High: VAEntrypointEncSliceLP`

### 試みた方法

ホストの `/usr/lib/x86_64-linux-gnu/dri/` (iHDドライバ) と `libva*.so.2` をコンテナにボリュームマウント。

```yaml
devices:
  - /dev/dri:/dev/dri
volumes:
  - /usr/lib/x86_64-linux-gnu/dri:/usr/lib/x86_64-linux-gnu/dri:ro
  - /usr/lib/x86_64-linux-gnu/libva.so.2:...
  - /usr/lib/x86_64-linux-gnu/libva-drm.so.2:...
```

### 結果

- ホストのlibvaをマウントしない場合: コンテナ内libva(v1.14)がホストのiHDドライバ(v1.17)を初期化できない (`has no function __vaDriverInit_1_0`)
- ホストのlibvaをマウントした場合: `vaapih264enc` は認識されるが、**映像が壊れる**。ABI互換性の問題と推測。

### 今後の方針

ホストと同じlibva/iHDドライババージョンを含むカスタムDockerfileをビルドする必要がある。restreamioイメージへのライブラリマウントでは安定動作しない。

## 現在の動作構成
