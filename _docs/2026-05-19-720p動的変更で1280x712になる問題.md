# 720p動的変更で1280x712になる問題

日付: 2026-05-19

## 背景

低画質設定で720pソースを使うようにしたところ、HDMIキャプチャまたは配信経路が720pとして `1280x712` を出しているように見えた。

直近の未コミット変更では、`/api/stream` で解像度変更を受けたときに、`scaler.caps` だけでなく `capture.caps` も変更するようになっていた。

該当箇所:

- `nxmc/cmd/nxmc-server/main.go`
  - `req.Height < 1080` のとき `capture.caps=image/jpeg,width=1280,height=720,framerate=30/1`
  - `scaler.caps=video/x-raw,width=<req.Width>,height=<req.Height>`
- `streaming/entrypoint.sh`
  - `v4l2src ! capsfilter name=capture ... ! jpegdec ! videoconvert ! videoscale ! capsfilter name=scaler ... ! vaapih265enc ... ! rtspclientsink`

## 最初の観測

gstdで720p状態になっているとき、`nxmc` コンテナ内からRTSPを `ffprobe` した。

```sh
docker compose exec -T nxmc ffprobe \
  -v error \
  -rtsp_transport tcp \
  -i rtsp://mediamtx:8554/cam \
  -select_streams v:0 \
  -show_entries stream=width,height,coded_width,coded_height,sample_aspect_ratio,display_aspect_ratio \

```

結果:

```text

height=712
coded_width=1280
coded_height=720
sample_aspect_ratio=1:1
display_aspect_ratio=160:89
```

`/api/screenshot` の出力も `1280x712` になった。

```sh

file /tmp/web-switch-shot-source.jpg

  -show_entries stream=width,height,coded_width,coded_height,sample_aspect_ratio,display_aspect_ratio \

```

結果:

```text

width=1280
height=712
coded_width=1280
coded_height=712
sample_aspect_ratio=1:1
display_aspect_ratio=160:89
```

この時点では「HDMIキャプチャが720pとして712行しか出していない」可能性と、「後段のH.265/RTSP/動的変更が表示クロップを壊している」可能性があった。

## gstd実行中グラフの確認

gstreamerコンテナ内でgstdのパイプライン状態を確認した。

```sh
docker compose exec -T gstreamer curl -sf http://127.0.0.1:5000/pipelines/cam
```

重要な観測:

- `v4l2src -> capture` は `image/jpeg width=1280 height=720`
- `jpegdec -> videoconvert -> videoscale -> scaler` は `video/x-raw width=1280 height=720`
- `scaler -> encoder` も `video/x-raw width=1280 height=720 format=NV12`
- `encoder -> h265parse -> rtspclientsink` のGStreamer graph上も `width=1280 height=720`

つまりGStreamer内部のcaps表示だけを見ると720pであり、`712` は見えない。

ただしRTSPを受けるffmpeg側では `coded_height=720`、表示 `height=712` になっている。

## 考察

今回の `1280x712` は、H.265の符号化サイズそのものが712行になる問題ではない。

根拠:

- 問題時も `coded_height=720`
- 表示 `height=712` だけが変わる
- GStreamer graph上のcapsは `1280x720`
- `720 - 712 = 8` は `1088 - 1080 = 8` と一致する

1080pでは `coded_height=1088`（CTU 64境界へのパディング）に対し conformance window で8ライン切り詰めて `height=1080` にしている。動的caps変更後の720pで同じ8ラインが切り詰められていることから、1080p時のSPS (Sequence Parameter Set) に書かれた `conf_win_bottom_offset=8` が、caps renegotiation後も更新されずに720pストリームに適用されていると考えられる。

720は64の倍数なのでパディング不要であり、conformance windowのcrop_bottomは0であるべき。

再現条件:

- 初期状態から720pで起動: 正常（SPSが最初から正しく生成される）
- ファイル出力: 正常（SPSが新規生成される）
- ffmpeg VAAPI: 正常
- gstdのPLAYING中に `capture.caps` / `scaler.caps` を1080pから720pへ変更: 異常（古いSPSが持ち越される）

このため、問題箇所は `vaapih265enc` 単体ではなく、gstd経由で再生中caps renegotiationを行う設計にある。

## gstd pipeline delete/createの検証

PLAYING中のcaps変更ではなく、パイプラインを削除して新しい解像度で再作成する方式を検証した。

`/api/stream` の実装を以下のように変更:

1. `PUT /pipelines/cam/state?name=null`
2. `DELETE /pipelines?name=cam`
3. 2秒待機（mediamtxのpublisher session解放待ち）
4. 新しいcapsで `POST /pipelines?name=cam&description=...`
5. `PUT /pipelines/cam/state?name=playing`

nxmcからgstd APIを呼ぶため、パイプラインdescription構築に必要な `AUDIO_DEVICE` 環境変数をnxmcコンテナにも渡した。

### 結果

coded_width=1280
coded_height=720
sample_aspect_ratio=1:1
display_aspect_ratio=16:9

# 720p → 1080p
width=1920

coded_width=1920

# 再度 720p

712問題は完全に解消。往復も正常。

### 残課題: WHEP再接続

pipeline delete/createではRTSPストリームが一度消えるため、ブラウザのWebRTC接続が切れる。現状のWHEP実装（`whep.js`）にはICE disconnect時の再接続ロジックがあるが、以下の問題が発生した:

- mediamtxの新stream登録前にWHEP再接続が走り `404` で失敗する
- ICE auto-reconnectと、changeStream後の明示的reconnectが競合してセッションが乱立する
- 成功した接続が後発のreconnectタイマーに殺される

pipeline再作成自体の所要時間は約4秒（null + delete + 2s待機 + create + play）。その後mediamtxがstreamを登録するまでさらに1-2秒。

この再接続制御を整理してから実装する。

### 実装メモ

`/api/stream` は、解像度変更時にgstdのcapsをPLAYING中に差し替えず、pipeline delete/createを実行する形へ変更した。ビットレートのみの変更は従来通り `encoder.bitrate` のプロパティ更新で済ませる。

ブラウザ側のWHEP再接続は `whep.js` に接続世代を持たせ、古いICE disconnectタイマーや古い再接続ループが後続の成功済み接続を閉じないようにした。`/api/stream` のレスポンスには `restarted` を含め、pipeline再作成が発生したときだけ `settings.js` から少し待ってWHEPを明示的に再接続する。明示再接続は1.5秒後に開始し、stream online前に失敗しても1秒間隔で再試行する。

nxmcがpipeline descriptionを再構築できるように、compose定義ではnxmcコンテナにも `WIDTH` / `HEIGHT` / `BITRATE` / `AUDIO_DEVICE` / `RTSP_HOST` を渡す。

## 未検証: NULL → READY → PLAYING

前回のNULL経由の検証では NULL → caps変更 → PLAYING と直接遷移して失敗した（`rtspclientsink` が再publishできず `invalid SDP: no media streams`）。

READY状態を挟めば、caps negotiationが完了してからPLAYINGに入れる可能性がある。

```
NULL → caps変更 → READY（caps negotiation完了を待つ）→ PLAYING
```

delete/createより軽量で、ストリーム断の時間も短くなる可能性がある。ただしrtspclientsinkがREADY→PLAYING遷移でRTSPセッションを正しく再確立できるかは未知数。

## 結論

- HDMIキャプチャが `1280x712` を出しているわけではない。
- ffmpeg VAAPIやGStreamer VAAPI単体の720pエンコードでも再現しない。
- 開始時から720pのGStreamer RTSP publishでも再現しない。
- `gstd` でPLAYING中のパイプラインに `capture.caps` / `scaler.caps` を差し替えると再現する。
- **gstd pipeline delete/createで712問題は解消することを確認済み。**

実装修正の方向性:

- `/api/stream` でgstdのcapsをPLAYING中に直接差し替えるのは避ける。
- 解像度変更は gstd pipeline delete/create で行う。
- WHEP再接続は接続世代で制御し、古い再接続タイマーが新しい接続を壊さないようにする。
